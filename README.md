# Тестирование API сервиса Wishlist

## 1. О проекте
В этом репозитории находятся материалы по тестированию REST API сервиса для управления списками желаний.
**Цель работы:** проверить основной функционал API, включая авторизацию, создание и управление вишлистами и подарками.
**Тестируемый сервис:** `https://api.wishlist.otus.kartushin.su`

## 2. Инструменты
*   **Postman** — для создания и выполнения коллекции тестов.
*   **SoapUI** — для дополнительной проверки и отладки (проект прилагается).

## 3. Структура репозитория
wishlist-api-tests/

├── postman/

## 4. Как запустить тесты (Инструкция)

### 4.1. В Postman
1.  Откройте Postman.
2.  Импортируйте коллекцию: **File → Import** → выберите файл `postman/Wishlist API QA Professional Suite.postman_collection.json`.
3.  **Никаких дополнительных окружений настраивать не нужно** — все переменные (`base_url`, `token`, `wishlist_id`, `gift_id`) уже определены внутри коллекции и будут автоматически заполняться в процессе выполнения тестов.
4.  Запускайте запросы последовательно в таком порядке:
    *   Сначала папка **`AUTH TESTS`** (регистрация → логин)
    *   Затем папка **`WISHLIST TESTS`**
    *   И наконец папка **`GIFT TESTS`**
    
    Обратите внимание: тесты в коллекции уже содержат скрипты, которые автоматически сохраняют токен после логина и ID созданных объектов в переменные коллекции (`pm.collectionVariables.set()`), поэтому следующие запросы будут использовать актуальные данные.

### 4.2. В SoapUI
1.  Откройте SoapUI.
2.  Выберите **File → Import Project** и укажите файл `soapui/wishlist-soapui-project.xml`.
3.  В проекте перейдите в тест-сьют `Wishlist API Test Suite` и запускайте тест-кейсы.

## 5. Тест-план
### 5.1. Объекты тестирования
Были выбраны основные эндпоинты для проверки ключевых пользовательских сценариев:

| Группа | Метод | Эндпоинт | Описание |
| :--- | :--- | :--- | :--- |
| **Аутентификация** | `POST` | `/api/auth/register` | Регистрация нового пользователя |
| | `POST` | `/api/auth/login` | Авторизация и получение токена |
| **Вишлисты** | `POST` | `/api/wishlists` | Создание нового списка желаний |
| | `GET` | `/api/wishlists` | Получение всех своих списков |
| | `GET` | `/api/wishlists/{id}` | Получение конкретного списка по ID |
| | `PUT` | `/api/wishlists/{id}` | Обновление данных списка |
| **Подарки** | `POST` | `/api/gifts/wishlist/{wishlistId}` | Добавление подарка в список |
| | `GET` | `/api/gifts/wishlist/{wishlistId}` | Получение всех подарков из списка |
| | `PUT` | `/api/gifts/{id}/reserve` | Резервирование подарка |

### 5.2. Сценарии тестирования
Для каждого эндпоинта проверялись позитивные (успешные) и негативные (ошибочные) сценарии.

#### Позитивные сценарии (Happy Path)
*   Регистрация нового пользователя с валидными данными.
*   Вход в систему с корректным логином и паролем.
*   Создание, получение и обновление вишлиста авторизованным пользователем.
*   Добавление подарка в вишлист с валидными `name` и `price`.
*   Резервирование созданного подарка и проверка изменения статуса `reserved`.

#### Негативные сценарии
*   Попытка повторной регистрации того же пользователя.
*   Вход с неверным паролем.
*   Доступ к защищенным эндпоинтам без заголовка `Authorization`.
*   Создание подарка без обязательного поля `price`.
*   Создание подарка с некорректной структурой JSON.

## 6. Результаты тестирования (Отчет)

### 6.1. Позитивные сценарии (PASSED)
Все проверенные позитивные сценарии завершились успешно, демонстрируя стабильную работу основного функционала API.

*   ✅ **Регистрация:** `POST /api/auth/register` → **200 OK**. Новый пользователь создается.
*   ✅ **Логин:** `POST /api/auth/login` → **200 OK**. Возвращается валидный JWT-токен.
*   ✅ **Создание вишлиста:** `POST /api/wishlists` → **200 OK**. Возвращается объект вишлиста с присвоенным `id`.
*   ✅ **Обновление вишлиста:** `PUT /api/wishlists/{id}` → **200 OK**. Данные вишлиста успешно изменяются.
*   ✅ **Создание подарка:** `POST /api/gifts/wishlist/{wishlistId}` → **200 OK**. Возвращается объект подарка с `id`.
*   ✅ **Резервирование подарка:** `PUT /api/gifts/{id}/reserve` → **200 OK**. В ответе поле `"reserved": true`, что подтверждает изменение состояния.

### 6.2. Негативные сценарии (PASSED)
API корректно отклоняет некорректные запросы.

*   ✅ **Повторная регистрация:** Запрос на регистрацию уже существующего пользователя вернул `401 Unauthorized`. Ожидаемое поведение.
*   ✅ **Неверный пароль:** Запрос на логин с правильным логином, но неверным паролем вернул `403 Forbidden`. Доступ заблокирован.
*   ✅ **Запрос без авторизации:** `GET /api/wishlists` без заголовка `Authorization` вернул `403 Forbidden`. Эндпоинт защищен.
*   ✅ **Невалидное тело запроса:** Отправка `POST /api/gifts/wishlist/{wishlistId}` с некорректным JSON (например, без поля `price`) вернула `403 Forbidden`. Входные данные валидируются.

### 6.3. Найденные проблемы и замечания
В ходе тестирования были обнаружены следующие проблемы и несоответствия:

#### 1. ⚠️ **НЕСООТВЕТСТВИЕ: Ответ при регистрации не соответствует спецификации**
*   **Описание:** При успешной регистрации (`POST /api/auth/register`) сервер возвращает статус `200 OK` и тело ответа в формате `text/plain` с сообщением "Пользователь успешно зарегистрирован". 
*   **Ожидаемое поведение (по Swagger):** Согласно спецификации OpenAPI, этот эндпоинт должен возвращать `200 OK` с телом типа `object` (скорее всего, JSON с данными пользователя или пустой объект).
*   **Фактическое поведение:** Возвращается текст, а не JSON. Это не ошибка выполнения (текст корректен), но **нарушение контракта API**, описанного в документации.
*   **Приоритет:** Низкий. Функционально запрос работает, но клиент, ожидающий JSON, может сломаться.

#### 2. ⚠️ **РЕКОМЕНДАЦИЯ: Неинформативные ответы с кодом 403**
*   **Описание:** В различных негативных ситуациях (неверный пароль, отсутствие токена, невалидное тело) сервер возвращает один и тот же код `403 Forbidden` с пустым телом.
*   **Рекомендация:** Для улучшения качества API (RESTful maturity) лучше использовать более специфичные HTTP-статусы и добавлять тело ошибки с пояснением причины:
    *   `401 Unauthorized` — для случаев отсутствия или неверной аутентификации.
    *   `400 Bad Request` — для невалидных данных в запросе.
    *   `403 Forbidden` — оставить для случаев, когда аутентификация пройдена, но у пользователя нет прав на действие.

В целом, API готов к использованию, но требует небольших доработок для повышения качества.
